# Rust Video Processing Web App Requirements

## Summary

构建一个面向普通创作者和站点维护者的 Rust Web 视频处理后台。第一版支持 2GB 级大文件上传、断点续传、服务端 FFmpeg 转码、转码后预览，并让用户用用途型预设导出 MP4 或 HLS/m3u8 分片包。

---

## Problem Frame

目标用户的视频来源不统一，常见问题是文件体积过大、格式不适合 Web 播放、移动端或浏览器兼容性不稳定，以及课程站和 Web 应用需要流媒体分片但原始拍摄文件没有 HLS 产物。用户希望用较少操作把视频整理成适合博客、课程站和 Web 应用分发的格式。

第一成功指标是操作简单。体积下降和播放稳定是质量门槛，但产品不应要求普通创作者理解完整转码参数才能得到可用结果。

---

## Key Decisions

- **Rust Web app as the core product.** 产品定位从 WebAssembly 应用调整为 Rust Web 应用，因为 2GB 大文件、断点续传、服务端队列和稳定 FFmpeg 转码更适合在服务端完成。
- **MP4 remains the default path.** 默认推荐 MP4 输出，适合个人博客、简单网页嵌入和直接下载使用；HLS/m3u8 作为高级发布选项出现。
- **HLS is creator-facing, not expert-facing.** HLS/m3u8 需要提供完整高级能力，但主入口仍然是用途型预设，参数细节可见但不压到主流程。
- **Single-user admin mode first.** 第一版以单用户或管理员模式运行，先支持历史任务、预览、下载和删除，不承担完整多租户账号体系。
- **Hard-coded safety limits first.** 第一版需要容量和并发保护，但不需要配置界面；超额时系统应阻止继续上传或排队转码。

---

## Actors

- A1. **管理员/创作者:** 上传原视频，选择预设，查看转码进度，预览结果，下载产物，删除历史视频释放空间。
- A2. **转码服务:** 接收上传任务，调用 FFmpeg 处理视频，保存产物，记录任务状态和错误。
- A3. **播放器预览:** 在转码完成后播放 MP4 或 HLS 产物，帮助用户确认画质、声音、比例和流媒体可播放性。

---

## Requirements

**Upload and task handling**

- R1. 系统必须支持单个视频文件上传，第一版单文件上限为 2GB。
- R2. 系统必须支持断点续传，上传中断后用户可以从已完成部分继续，而不是重新上传整个文件。
- R3. 系统必须在创建转码任务前检查容量和进行中任务上限，第一版总存储上限为 200GB，同时转码任务上限为 2 个。
- R4. 系统必须展示任务状态，让用户知道任务处于上传中、等待中、转码中、完成或失败。

**Presets and outputs**

- R5. 系统必须提供常见用途型预设，例如博客、课程、移动端和高清留档。
- R6. 每个预设必须能展示关键参数摘要，例如输出格式、分辨率、码率倾向和是否生成 HLS 分片。
- R7. 系统必须默认推荐 MP4 输出，让普通创作者以最少决策得到可下载视频。
- R8. 系统必须支持输出 MP4 文件，作为第一版最小价值路径。
- R9. 系统必须支持输出 HLS/m3u8 分片包，作为高级发布路径。
- R10. 用户必须能在 MP4 和 HLS/m3u8 之间选择输出目标。

**Preview, download, and retention**

- R11. 转码完成后，系统必须提供在线播放预览。
- R12. 预览必须覆盖 MP4 产物和 HLS/m3u8 产物，便于用户确认浏览器播放效果。
- R13. 系统必须提供产物下载；MP4 作为单文件下载，HLS/m3u8 作为包含播放列表、分片文件和简单说明的 zip 压缩包下载。
- R14. 系统必须保留上传原视频、历史任务和转码产物，直到管理员手动删除。
- R15. 管理员必须能删除历史任务及相关文件，以释放存储空间。
- R16. 系统必须显示存储占用或容量风险，让管理员知道何时需要清理历史视频。

**User experience**

- R17. 主流程必须让非技术用户在少量步骤内完成上传、选择预设、等待处理、预览和下载。
- R18. HLS/m3u8 的高级能力必须以用途预设为主呈现，避免把分辨率、码率和分片时长作为普通用户的必填决策。
- R19. 转码失败时，系统必须给出可理解的失败状态，并允许用户重新尝试或删除失败任务。

## Key Flows

- F1. MP4 preset conversion
  - **Trigger:** 管理员上传一个原始视频并选择默认推荐的 MP4 输出。
  - **Actors:** A1, A2, A3
  - **Steps:** 用户上传视频；系统完成断点续传；用户选择用途预设；系统创建转码任务；转码完成后用户预览 MP4；用户下载 MP4 文件。
  - **Outcome:** 用户得到一个适合博客或简单网页嵌入的压缩 MP4 文件。
  - **Covers:** R1, R2, R5, R7, R8, R11, R13, R17

- F2. HLS advanced publishing
  - **Trigger:** 管理员为课程站或 Web 应用选择 HLS/m3u8 输出。
  - **Actors:** A1, A2, A3
  - **Steps:** 用户上传视频；系统展示用途预设和关键参数摘要；用户选择 HLS 输出；系统生成 m3u8 与分片文件；用户预览 HLS 播放；用户下载 HLS 文件包。
  - **Outcome:** 用户得到可部署到站点或对象存储中的 HLS 分片包。
  - **Covers:** R5, R6, R9, R10, R12, R13, R18

- F3. Capacity protection and cleanup
  - **Trigger:** 系统达到容量保护线或进行中任务上限。
  - **Actors:** A1, A2
  - **Steps:** 用户尝试上传或创建任务；系统检测超过保护线；系统阻止新任务；管理员查看历史任务；管理员删除不再需要的视频和产物。
  - **Outcome:** 系统避免磁盘或转码队列失控，管理员可以通过清理恢复可用容量。
  - **Covers:** R3, R14, R15, R16

## Acceptance Examples

- AE1. **Covers R1, R2.** 给定用户上传一个接近 2GB 的视频，当网络中断后恢复上传，系统继续上传未完成部分，并最终允许该文件进入转码流程。
- AE2. **Covers R7, R8, R17.** 给定用户没有调整高级参数，当用户选择默认推荐路径，系统生成可预览和下载的 MP4 文件。
- AE3. **Covers R9, R12, R13.** 给定用户选择 HLS/m3u8 输出，当转码完成，系统提供浏览器内 HLS 预览和可下载的 HLS 文件包。
- AE4. **Covers R3, R15, R16.** 给定系统已达到容量保护线，当用户尝试上传新视频，系统阻止上传并引导管理员删除历史任务释放空间。
- AE5. **Covers R19.** 给定 FFmpeg 转码失败，当任务进入失败状态，用户可以看到失败状态并重新尝试或删除任务。

---

## Success Criteria

- S1. 非技术用户可以在少量步骤内完成一次 MP4 转码、预览和下载。
- S2. HLS/m3u8 输出不要求普通用户填写专业参数即可完成。
- S3. 2GB 级文件上传在中断后可以恢复，不需要重新上传整个文件。
- S4. 系统不会因为历史产物默认保留而无限制占满磁盘或堆积转码任务。
- S5. 输出产物能在浏览器预览中完成基本播放验证。

---

## Scope Boundaries

**Deferred for later**

- 公开托管播放链接和可嵌入播放器。
- 完整容量、并发和预设管理界面。
- 多用户账号、团队协作、权限管理和项目级素材库。
- 更复杂的批量上传和批量转码工作流。

**Outside first-version identity**

- 面向专业视频工程师的 FFmpeg 全参数控制台。
- 纯浏览器本地 WASM 转码作为核心处理链路。
- 课程内容管理系统或博客发布系统本身。

---

## Dependencies and Assumptions

- D1. 服务端可以使用 FFmpeg 完成 MP4 压缩转码和 HLS/m3u8 分片生成。
- D2. 第一版部署环境有足够磁盘空间承载原视频、历史任务和转码产物。
- D3. 第一版面向单用户或管理员使用，因此不需要处理多租户隔离和团队权限。
- D4. 用途型预设能够覆盖普通创作者的主要发布场景；专业参数作为可见细节或后续增强，不作为主流程必填项。

## Outstanding Questions

**Deferred to planning**

- OQ1. 断点续传的具体协议和前端交互细节。

- OQ3. 转码失败信息应该暴露到什么详细程度，才能帮助用户行动但不泄漏过多底层噪音。
